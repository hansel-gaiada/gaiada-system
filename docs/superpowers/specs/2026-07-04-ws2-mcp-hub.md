# Workstream 2 — MCP Hub

**Date:** 2026-07-04
**Status:** Design draft (brainstorming stage — not being built yet)
**Parent:** `2026-07-04-gaiada-ai-platform-roadmap.md` (Workstream 2)
**Depends on:** WS1 core schema + module framework + RBAC engine; WS3 Gateway (AI egress).
**Scope:** The integration layer exposing company data + tools to AI clients/surfaces via the MCP protocol. The "hub for all company tools" from the architecture diagram.

---

## 1. Role & boundary

- **MCP = access** (nouns & verbs): read company data, perform company actions. It is the one authoritative tool/data interface for AI.
- **Boundary vs Gateway:** **MCP exposes company tools/data; the Gateway (WS3) is the AI-provider egress.** An MCP tool that needs AI (OCR, Magnific, LLM) **calls the Gateway** — matching the diagram's MCP → Gateway → Third Party flow.
- **Clients:** WA bot, user-facing AI, custom agents, N8N automations.

---

## 2. Architecture — DECIDED

- **Single aggregating MCP server, federation-ready.** One server (official **MCP TypeScript SDK**), **co-located with each platform instance**, surfaces: **core tools + each module's registered `mcpTools` (via `ModuleContract`) + external-tool wrappers**. Can split into federated downstream MCP servers later if tools move to separate services.
- **Deployment: per-site hub + central hub.** Each company site runs a local MCP fronting its local platform data (low latency, offline-capable); the central hub adds cross-company/management tools. **Same codebase; scope set by deployment + RBAC.**

---

## 3. Transport & connectivity

- **Streamable HTTP** (MCP remote transport) — clients are networked across sites.
- Runs entirely on the **zero-trust floor**: mTLS, peer allowlist, no public listeners (reachable only via the sanctioned path). See WS7.

---

## 4. Fronts the platform — never bypasses it

- MCP tools invoke the **platform service layer** (NestJS services), so all business logic, validation, and RBAC apply. **No direct DB access from the hub.**
- Co-located → in-process/service calls to the local platform instance.

---

## 5. Authorization — on-behalf-of (OBO)

**The critical design point.** Authorization is on behalf of the **end user**, not the calling service.

1. Client authenticates to the hub with a **service identity** (mTLS + token).
2. Client **propagates the end-user principal** (e.g., WA bot resolves sender → user via `identity_links`; a UI passes the user's IdP token).
3. The hub assembles the principal and calls **Cerbos** to:
   - **filter tool visibility** — advertise only tools the user may use;
   - **authorize each call** against its target resource + action.
4. **Unknown/unauthenticated end user → minimal principal** → only public/general tools (consistent with WA bot Phase 3 + the RBAC spec §5.2).

This prevents privilege confusion (the bot's service identity never grants the bot broader data access than the actual user has).

---

## 6. MCP primitives exposed

- **Tools (verbs):**
  - Core: `projects.list/get/create/update`, `tasks.*`, `clients.*`, `deliverables.*`, `time.log`, `search` (RAG over stored data), `activity.feed`, `rollup.metrics` (management/`group_executive`).
  - Module (example, agency): `agency.listCampaigns`, `agency.campaignStatus`, `agency.pendingApprovals`, `agency.createBrief`.
  - **External/AI-backed (wrap the Gateway):** `ocr.extract`, `vision.describe`, `media.transcribe`, `image.enhance` (Magnific), `llm.summarize`.
- **Resources (readable context):** key entities exposed for attach-as-context reads.
- **Prompts (optional):** reusable templates for common tasks (e.g., "summarize project status").

Tools are contributed via `ModuleContract.mcpTools`; core + an integrations module register the rest. A **tool registry** aggregates and advertises them (filtered per-request by RBAC).

---

## 7. Scope by deployment

| Hub | Exposes |
|---|---|
| **Per-site** | Its local tenant's data + action tools; local AI-backed tools via the local Gateway. |
| **Central** | Cross-company/management tools (rollups, `group_executive` queries) over the aggregated data. |

---

## 8. Safety & ops

- **Every tool call audited** (via `activities`) — who, tool, args (redacted as needed), decision, result.
- **Rate-limited** per client/principal.
- **Destructive / high-impact tools flagged** — may require confirmation, and integrate with the WS7 tiered-autonomy model.
- **Tools versioned + discoverable** via MCP list; **schema-tolerant** for rolling upgrades.
- Loop-guarding is the calling agent's responsibility; the hub enforces per-call authorization regardless.

---

## 8b. D16 — set-returning authorization (LOCKED, adversarial review)

Per-resource authz on set-returning tools (`projects.list`, `search`, `activity.feed`) is an N+1 (N Cerbos evals per call × agent fan-out). Instead: use **Cerbos `PlanResources` (query-planning)** for all set-returning tools → the decision becomes a predicate **pushed into SQL/RLS** (O(1) authz per list). Reserve per-resource `CheckResources` for single-gets/mutations. Apply the same plan predicate as the **RAG pre-filter** (D9). Cache decisions keyed on `principal + updated_at`.

## 9. Open items
- Exact OBO token format + propagation (IdP token vs signed principal assertion from a trusted surface).
- Which reads are Tools vs Resources (context ergonomics).
- Central-hub read path: query central DB directly vs fan-out to site hubs (latency vs freshness).
- Tool argument validation/schema standard (zod → JSON Schema).
- Pagination/streaming conventions for large result sets.
- Rollout of external-tool wrappers as the Gateway (WS3) firms up.
